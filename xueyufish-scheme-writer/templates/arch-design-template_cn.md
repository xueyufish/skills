# 1. 需求定义

## 1.1 原始需求
概要描述原始需求的内容，不需要详细描述。可以通过链接的方式对原始需求文档进行引用。

## 1.2 需求来源
描述需求的来源，如客户需求、产品演进、研发规划等。

## 1.3 需求背景
描述项目的背景、目标和意义，说明需求的业务价值或技术价值。

## 1.4 需求目标
描述项目的核心目标，如：功能、性能、用户体验等。列出的目标需要是可量化，并且能够通过指标进行评估的。

## 1.5 需求范围
描述项目的功能范围、技术范围和交付范围。对于不包含在项目范围内的功能，需要明确说明。

## 1.6 需求约束
* 时间约束：项目周期及关键里程碑
* 资源约束：人力、技术栈、部门/团队约束等
* 技术约束：项目需要使用的技术、工具、框架等。
* 法律约束：项目需要遵守的法律、法规、政策等。

## 1.7 术语
解释需求、方案中涉及的专业术语。以表格的方式列出中文术语、英文全程、英文缩略语、中文解释。例如：

| 中文术语 | 英文全程 | 英文缩略语 | 中文解释 |
| -------- | -------- | ---------- | -------- |
| 功能需求 | Functional Requirement | FR | 系统需要实现的功能 |
| 非功能需求 | Non-functional Requirement | NFR | 系统对性能、安全、可靠性等方面的要求 |
| 场景 | Scenario | S | 系统在特定条件下的行为 |
| 用例 | Use Case | UC | 系统的一个功能场景 |
| DFX 属性 | DFX Attribute | DFX | 系统的性能、安全、可靠性等方面的属性 |

# 2. 需求分析

## 2.1 需求概述
概要描述需求的整体流程、需求说明。

## 2.2 功能需求分析 
分场景、用例列出对功能需求的分析，建议通过用例图、用例规约的方式。在每个场景中，通常需要包含以下内容：

1. **正常场景**：描述系统在各种正常场景的处理情况。场景应简明、完整、不含糊、可验证、必要的。至少包括：前置条件、触发事件、关键步骤、成功保证（后置条件）四部分内容。需要描述清楚在什么条件下，发生什么事件后，系统如何处理，最后系统输出什么等内容。
2. **异常场景**：描述系统在各种异常场景中的处理情况。场景应简明、完整、不含糊、可验证、必要的。至少包括：前置条件、触发事件、关键步骤、后置条件四部分内容。需要描述清楚在什么条件下，发生什么异常事件后，系统如何处理，最后系统输出什么等内容。
3. **DFX 属性**：描述此功能需求的 DFX 属性。例如：性能、安全、可靠性等。

### 2.2.1 场景1分析
此处描述场景1的分析内容，包括前置条件、触发事件、关键步骤、成功保证（后置条件）等。

### 2.2.2 场景2分析
此处描述场景2的分析内容，包括前置条件、触发事件、关键步骤、后置条件等。

## 2.3 非功能需求分析
分性能、安全、可靠性/可用性、可维护性等维度进行非功能分析。

### 2.3.1 性能分析
描述关于系统的性能指标相关分析，如响应时间、并发用户数、吞吐量、大表治理策略等

### 2.3.2 安全分析
描述关于系统的安全指标相关分析，如数据加密、权限控制、操作日志、密钥、对外服务中的加密、数据安全等。

### 2.3.3 可靠性 / 可用性分析
描述关于系统的可靠性和可用性指标相关分析，如系统可用性、容错机制、备份与恢复等。

### 2.3.4 可维护性分析
描述关于系统的可维护性指标相关分析，如日志记录、扩展性、维优能力、告警能力等。

## 2.4 E2E 验收标准
明确需求的端到端验收条件和验收方式。

## 4.3 API 设计
定义需求涉及新增、修改、删除的 API 接口描述。

### 4.3.1 用户面接口设计
指面向前台客户的接口和面向后台用户的接口。前台用户指面向用户侧 APP、H5、PC 等访问系统的用户，后台用户指面向系统管理员、开发人员等访问系统的用户。

前台客户接口和后台用户接口需要分开设计，确保权限隔离。

### 4.3.2 内部接口设计
指微服务之间调用的接口，不向页面等其他终端开放。

内部微服务间 API 接口，需要标注清楚调用方、使用场景、预估响应时间、预估吞吐量、流程策略。

### 4.3.3 开放接口设计
指提供给企业、服务商的开放接口，用于与其他系统进行集成。

开放接口设计过程中充分考虑兼容性问题，需要在接口设计中考虑到不同系统之间的差异，如数据格式、协议版本等。

### 4.3.4 运维面接口设计
指提供给运维人员使用的接口，用于监控、管理、维护系统。

### 4.3.5 其他接口设计
指其他未分类的接口，如健康检查接口、指标采集接口等。

## 4.4 消息设计
指 Kafka、RabbitMQ 等微服务之间用于通信的消息。默认情况下如无特殊说明使用 Kafka 消息队列。

设计时需要包含以下内容：
- Topic：消息的主题名称
- 消息提供方：消息的发送方，即生产消息的微服务
- 消息消费方：消息的接收方，即消费消息的微服务
- 使用场景： 消息的使用场景
- 消息参数：JSON 格式的消息定义
